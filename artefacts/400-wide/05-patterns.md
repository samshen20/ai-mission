# 05-patterns.md — Pattern Placement on L2 (Meridian Phase 1)

Each pattern is mapped to the specific containers and relationships in
`02-containers.mmd`. "Where on L2" names edges or containers from that diagram.
"Meridian constraint" cites the brief, ADR, or discovery finding that forced
the pattern. "Trade-off" names the concrete cost Meridian pays for it.

| Pattern | Where on L2 | Meridian constraint it addresses | Trade-off |
|---|---|---|---|
| **Strangler Fig** | Apollo GraphQL Gateway routes per-endpoint: `/cart/*`, `/checkout/*` → Meridian Cart/Checkout (gRPC); `/catalog/*`, `/search/*` → legacy country-site backend until PIM migration completes. Web App (Next.js) and Mobile App (React Native) are per-country repointable BFFs. | "No Big Bang cutover" — 22 country sites must migrate independently; pilot country goes live while 21 remain on legacy. | Apollo Gateway becomes an undeletable routing layer — every legacy route must be maintained until its country migrates. Can't remove a route without knowing every country's migration status. |
| **Outbox** | Cart Service + Checkout Service + Identity Service → local Outbox table (same DB transaction) → Order Event Bus (Apache Kafka, `Kafka/TCP`). Three Rel edges: `cart → eventbus`, `checkout → eventbus`, `identity → eventbus`. | Cross-service consistency without distributed transactions (ADR-002). After service extraction, Cart and Checkout will be independent processes — the Outbox ensures no event is published without its state change committed, and no state change is committed without its event eventually published. | Adds an Outbox table per service and a relay process per service. If the relay crashes, events accumulate in the Outbox table — consumers see stale state until replay catches up. Kafka broker becomes a critical production dependency before a single cart is created. |
| **Circuit Breaker** | Three external edges: `checkout → stripe` (JSON/HTTPS), `gateway → auth0` (OAuth 2.0 / OIDC), `identity → auth0` (OAuth 2.0 / OIDC). Internal gRPC edges (Gateway → Cart/Checkout/Identity) are in-process during Phase 1 — circuit breaker NOT applied to them. The `checkout → stripe` circuit breaker SHALL be per-payment-method (card, iDEAL, SEPA, Klarna each have independent error-rate tracking and independent open/closed state). The Checkout Service SHALL expose `payment_methods: [available_methods]` in the cart response so the Web App BFF can disable only the degraded method's button. | PSD2 SCA latency budget (1,500ms). Stripe and Auth0 degrade independently of Meridian; a partial Stripe degradation (card payments failing, iDEAL healthy) must not block healthy payment methods. A global circuit breaker on `checkout → stripe` turns a single-payment-method degradation into a total checkout outage. | Circuit thresholds must be tuned per dependency AND per payment method: Stripe card payments permit higher error rate (10% threshold), iDEAL permits lower (5% — fewer transactions, each more valuable). Half-open probe interval = 15s per method. A misconfigured Auth0 circuit that opens on transient DNS blip denies all logins. |
| **BFF** | Web App (Next.js) and Mobile App (React Native) each front the Apollo GraphQL Gateway (`JSON/HTTPS`). Web BFF requests full availability payloads (qtyOnShelf + sellThroughToday + freshness); Mobile BFF requests compact payloads (pre-computed totals, offline-ready cart). POS browser is served by the Web App BFF. | 12 mobile apps + 22 country sites with diverging payload size and latency needs. A single API for all channels forces mobile over cellular to receive web-sized payloads. | Three codebases (Web BFF, Mobile BFF, the shared Apollo Gateway) — a field added to the Gateway schema must be opted into by each BFF. Divergent BFFs can silently serve different data for the same SKU if one lags on a schema update. |
| **Pipe & Filter** | Outbox relay (filter: serializes domain events) → Order Event Bus / Apache Kafka (pipe: partitioned topic, `Kafka/TCP`) → future extracted service consumer (filter: projection logic). The pipe is the Kafka topic; the filters are the producer's serialization and the consumer's projection. | Service extraction without producer changes — a new Loyalty Service can consume `OrderPlaced` from Kafka without Cart or Checkout modifying a single line of code. | Filter stages are independent — if the consumer projection crashes, the pipe buffers events but the read model falls behind. No back-pressure from consumer to producer in the current topology. |
| **CQRS** | Inventory Read Cache (Redis, `ContainerDb`) is the read model optimized for Cart Service queries (`RESP/TCP`). SAP ECC (`System_Ext`) is the write model — pushes batch files via `SFTP/IDoc, batch 15-30 min`. Cart Service reads from Redis; never writes to it. | SAP batch reality — inventory data is 15–30 min stale by design. The CQRS separation makes the staleness explicit: the write path is slow, batch-oriented, and external; the read path is fast, query-optimized, and internal. No synchronous SAP call during a cart read. | The read model is only as fresh as the last SAP batch. If the batch is delayed (SAP ECC outage, SFTP failure), the Inventory Read Cache silently serves stale data. The architecture accepts this staleness — it's better than a synchronous SAP call that would blow the PSD2 SCA budget. |

## Patterns recommended but not yet placed on L2

| Pattern | Why recommended | What's missing on L2 today |
|---------|----------------|--------------------------|
| **Bulkhead** | Multi-country rollout (post-pilot) will need per-region resource isolation — a traffic spike in France must not exhaust Germany's connection pool. | L2 shows a single monolithic deployable. No per-region pods, no separate connection pools, no resource quotas. Bulkhead is not applicable during Phase 1 pilot (single country, single deployable). Recommend adding a `Container` for each regional API Gateway instance with per-region connection pools when the L2 is updated for multi-country Phase 2. |
| **Saga** | Checkout spans Cart → Stripe → SAP ECC — after service extraction, this becomes a distributed transaction needing compensating actions (refund on SAP rejection, cart restoration on payment failure). | L2 shows Cart and Checkout as modules inside a single deployable — the checkout flow uses a local database transaction, not a distributed Saga. Saga is not applicable during Phase 1. Recommend adding a `CheckoutSaga` container or annotating the `OrderPlaced` / `OrderCancelled` events as Saga participants when the L2 is updated for post-extraction Phase 2. |

## User-suggested patterns that do not map to Meridian L2

| User example | Why it doesn't map |
|-------------|-------------------|
| Bulkhead: "Per-payment-method gateway pods, separate connection pools" for Postepay / PayPay / Klarna | Meridian routes payments through Stripe (`checkout → stripe`), not through per-method gateway pods. Stripe handles local payment method diversity — Meridian does not operate payment-method-specific infrastructure. If Meridian later adds direct integrations for payment methods Stripe doesn't support, Bulkhead per payment method becomes applicable. |
| Outbox: "CDC / Debezium" | ADR-002 explicitly forbids CDC: "Do NOT use Kafka Connect or Debezium to capture PostgreSQL WAL changes — application-level domain events are the stable API between services; CDC couples every downstream consumer to the current database schema." Meridian's Outbox uses application-level relay, not CDC. |
| Strangler Fig: "routes /v1/* to legacy, /v2/* to commercetools" | ADR-001 explicitly rejects commercetools: "Do NOT introduce commercetools, Elastic Path, or any headless-commerce SaaS." Meridian routes to its own modular monolith, not to a SaaS commerce engine. |
