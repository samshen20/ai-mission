# 04-tradeoff-matrix.md — Constraint × Approach Matrix

**Decision:** Build, not buy. Meridian builds the headless commerce platform.
**Rows:** Three architecture approaches from `00-options.md`.
**Columns:** Five load-bearing constraints from the Meridian brief and
`00-discovery-context.md`.

Each cell is scored with a one-line reason — no standalone numbers.

---

| Constraint →<br/>Approach ↓ | **PSD2 SCA latency**<br/>Payment auth must complete within SCA timeout window | **SAP batch-update reality**<br/>Inventory data is 15–30 min stale by design | **Junior-team operability**<br/>Can a non-senior team build, debug, and run this? | **Strangler-fig coexistence**<br/>22 countries migrate independently, no Big Bang | **18-month / $42M budget**<br/>Can the approach deliver a pilot country within the constraints? |
|---|---|---|---|---|---|
| **A: Modular Monolith** | **Lowest latency.** Payment orchestration is in-process — no cross-service network hop between cart and checkout. The Stripe API round-trip is the only external cost. SCA timeout risk is the Stripe→bank leg, which is identical across all three approaches. | **Natural fit.** The monolith's cart module reads SAP-populated cache synchronously with no impedance mismatch. Freshness indicator logic is co-located with cart — no translation layer between batch data and query path. | **Easiest to staff and operate.** One process, one log, one deployment pipeline. A mid-level engineer can trace a bug from the API gateway through to the database without crossing service boundaries. On-call is a single runbook. | **Worst fit.** One deployable means one release cadence for all countries. France's hotfix redeploys Germany's unchanged adapter. A broken release blocks every country's migration simultaneously. The monolith optimises for unification at the exact point where the migration needs per-country isolation. | **Fits.** 8–12 engineers for 18 months at Meridian-scale salary + infrastructure is within $42M. The monolith doesn't consume budget on Kafka clusters, schema registries, or distributed tracing before the first cart is created. Pilot country live in 3–6 months. |
| **B: Event-Sourced CQRS** | **Adds risk under load.** Payment state propagates through the event log asynchronously — the read model may not reflect `PaymentAuthorised` until the event is consumed and projected. If SCA timeout is tight (sub-second), the projection lag window adds uncertainty. Mitigable with a synchronous checkout→Stripe call inside the event handler, but this undercuts the "no sync calls" premise. | **Architectural mismatch.** SAP delivers batch files, not event streams. Every batch dump must be translated into synthetic events (`InventorySnapshotReceived`) to enter the log. The architecture's core abstraction — real-time event stream as truth — is built for a data source that does not emit events. The projection is no fresher than Approach A, but the infrastructure to get there is an order of magnitude more complex. | **Hardest to staff and operate.** Kafka cluster management, schema registry, consumer-lag alerting, poison-pill replay, event versioning — production concerns before the first cart exists. A junior engineer debugging why the loyalty projection is 47 events behind at 3am cannot reason about it under incident pressure. Requires 3–5 senior distributed-systems engineers Meridian may not have. | **Best fit.** Legacy and new systems coexist as event producers on the same log — no synchronous coupling, no dual-write problem. Migration is per event type, not per country. Cutover happens when new services consume the full event stream and legacy producers are turned off. Cleanest strangler pattern of the three. | **Does not fit.** 15–25 engineers + platform team + Kafka infrastructure + 9–15 months to first pilot. $42M may be consumed by platform build before a single country migrates. The event-sourced infrastructure must be built, tested, and hardened before commerce logic runs — the budget timeline compresses the learning curve the team needs. |
| **C: Buy Primitives, Build Integration** | **Adds one BFF hop.** SCA flow is BFF → Stripe → bank → BFF. The BFF adds ~5–10ms vs a direct call, negligible against the bank's SCA redirect. Stripe handles PSD2 compliance — Meridian doesn't build or maintain SCA logic. Lowest implementation risk for PSD2. | **Hardest mismatch.** Every bought SaaS primitive (Stripe, commercetools, loyalty SaaS) assumes real-time data. Meridian's SAP inventory is batch. A customer pays in real-time for an item whose shelf availability was last confirmed 22 minutes ago. The BFF must reconcile two incompatible time models, and no vendor in the stack understands the gap. The integration layer Meridian builds is the only place this mismatch is visible. | **Middle ground.** Operations are partly offloaded to SaaS vendors (Stripe runs payments, Auth0 runs auth), but the integration surface is wide. Debugging why a Stripe webhook wasn't delivered or why Auth0 token validation failed requires vendor-specific expertise. Easier than operating Kafka; harder than operating one process. | **Good fit.** Per-endpoint routing in the BFF — `/cart/add` routes to the new backend when ready, `/catalog/search` still hits legacy. Migration granularity is a single API endpoint. Clean, but depends on BFF routing discipline: every endpoint must be individually migration-tested, and route misconfiguration silently breaks the customer experience. | **Tight fit.** Fastest to pilot (6–12 weeks), smallest team (6–10 + vendor management). SaaS costs are negligible at pilot scale but grow linearly with traffic — at 22-country volume, the per-request Stripe/commercetools bill becomes a material cost-of-revenue line item not captured in the $42M capital budget. The budget covers build + initial SaaS; the operating model may force a re-budget at scale or a second migration off the SaaS. |

---

## Summary

| Constraint | Best fit | Worst fit |
|-----------|----------|-----------|
| PSD2 SCA latency | **A** — in-process, fewest hops | B — projection lag risk under tight SCA windows |
| SAP batch reality | **A** — no impedance mismatch | C — real-time SaaS vs batch inventory |
| Junior-team operability | **A** — one process, one runbook | B — requires senior Kafka/event-sourcing engineers |
| Strangler-fig coexistence | **B** — event log coexistence, per-event-type cutover | A — one release couples all 22 countries |
| 18-month / $42M budget | **A** — fits within timeline and budget for pilot | B — platform build consumes budget before migration begins |
