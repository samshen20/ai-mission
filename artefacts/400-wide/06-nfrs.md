# 06-nfrs.md — 7 NFR Budgets with Anti-Patterns

| # | NFR | Family | Container(s) | Target | Test approach | Meridian justification | Anti-pattern that would silently violate it |
|---|---|---|---|---|---|---|---|
| 1 | Cart-to-checkout p95 latency | Latency | Apollo Gateway → Cart Service → Checkout Service | p95 < 1,500ms (EU, incl. PSD2 SCA) | k6 synthetic checkout from pilot-region staging nightly; assert p95 ≤ 1,500ms across 1,000 iterations at 10 VUs; run on PR merge to release branch | "PSD2 SCA for EU payments" — SCA challenge round-trip dominates; the checkout path must complete before the bank's SCA window expires or the payment is declined and the cart is abandoned | Adding a synchronous SAP ECC `checkStockRealtime` call inside the Checkout path on every item. A single cache miss triggers a 400ms IDoc lookup; three misses in one cart push the total to ~1,450ms, breaching the 1,500ms ceiling with zero headroom for the Stripe→bank SCA leg. The Gateway health check passes, p50 looks fine, and the breach only appears at p95 — invisible to median-only dashboards. |
| 2 | Black Friday peak RPS | Reliability | Apollo Gateway, Cart Service, Checkout Service | 8,000 RPS sustained, 12,000 RPS burst (10 min) | k6 load test on PR merge to release branch; 8,000 RPS steady state for 30 min, ramp to 12,000 for 10 min, assert p99 latency < 2× baseline and zero 5xx; gate the merge if any container breaches | "Black Friday 2024 had a 40-minute outage in the EU" — a repeat outage during the pilot country's first Black Friday on the new platform destroys stakeholder confidence and triggers a rollback to legacy | Single-threaded Node.js event loop with no request queuing or backpressure. During burst, the event loop saturates on gRPC response deserialization; all incoming requests queue behind a single slow Cart Service call. The load balancer health check (`GET /health`) returns 200 OK instantly because it checks the process, not the event-loop lag — traffic keeps routing to a saturated instance while p99 latency diverges to infinity. The Gateway never returns 5xx so the error budget looks green. |
| 3 | Inventory accuracy | Quality | Inventory Read Cache (Redis), Order Event Bus (Kafka stock-event topic) | < 2% phantom-stock cancellation on click & collect (down from ~7% baseline) | Daily reconciliation script: compare `qtyOnShelf` in Inventory Read Cache against SAP ECC batch file for the same store+SKU+timestamp. Flag any store+SKU where cache value differs from SAP value by > 0. Assert: flagged count / total store+SKU pairs < 2%. Run nightly; alert if > 2% for 2 consecutive days | "~7% of click & collect orders cancelled at pickup due to phantom stock" — the binary 'In stock' badge on the existing 22 country sites is the root cause; the freshness indicator (Phase 0a) must drive this below 2% within 90 days of pilot launch | Hydrating the Inventory Read Cache lazily on the first cart read for a SKU. After a cache restart or TTL expiry, the first shopper who views that product triggers a synchronous SAP ECC fallback that populates the cache — but if SAP ECC is slow or the shopper clicks "Reserve" before the cache write completes, the Cart Service serves `availability: "unknown"` which the BFF renders as a greyed-out item. The shopper assumes "unavailable" and abandons the cart. If the item was actually on the shelf, this is a phantom-stock cancellation caused by the cache-warm path, not by stale data — the opposite of the freshness indicator's intent. |
| 4 | Platform availability | Reliability | Apollo Gateway + Checkout Service + Auth0 (SLA dependency) | 99.95% per region per month, excluding planned maintenance windows (announced ≥ 24h in advance) | Uptime monitor: synthetic probe (`lookupCustomerCart` + `checkoutSubmit`) from 3 geographic locations every 60s. SLO dashboard: error budget burn-rate alert at 99.92% (triggers P2). Planned maintenance windows pre-logged and excluded from denominator | "1,400 POS systems — no acceptable downtime window; stores must keep selling." A checkout outage during store operating hours means physical stores cannot transact. 99.95% permits ≤ 21.6 minutes of unplanned checkout downtime per month | Auth0 JWKS endpoint called synchronously on every Gateway request with `cache: false` or `max-age: 0`. When Auth0's JWKS endpoint experiences a transient 3-second degradation, every request to the Apollo Gateway hangs for 3 seconds waiting for the JWKS fetch. The Gateway health check (`GET /healthz`) returns 200 because the process is alive and the port is bound — the load balancer keeps routing traffic. Within 30 seconds, all Gateway worker threads are blocked on JWKS, the error budget burns to zero, and the platform is unavailable — but no alert fires because the health check never fails. Root cause: Auth0 is external but treated as in-process; the circuit breaker (NFR-applicability, `05-patterns.md`) was configured on `identity → auth0` (Management API) but NOT on `gateway → auth0` (JWKS). |
| 5 | Cost per checkout transaction | Cost | All Phase 1 hot-path containers + Stripe fees (excluded from target, tracked separately) | < $0.04 / completed checkout (compute, cache, database, event bus, bandwidth; excl. Stripe payment-processing fees) | Daily cost-allocation report from AWS CUR (Cost & Usage Report), broken down by container tag. Compute cost = (container vCPU-hours × $/vCPU) / checkout_count. Cache cost = (Redis node-hours × $/node-hour) / checkout_count. Assert: 7-day rolling average < $0.04. Alert if trending above $0.05 | "$42M over 18 months" + Meridian's digital share of ~31% of group revenue — unit economics on every checkout determine whether the platform's operating cost is sustainable when scaled from 1 pilot country to 22 | Logging full request and response bodies at INFO level for every checkout mutation. A single checkout produces: the GraphQL mutation body (~2KB), the Cart Service gRPC response with enriched availability for N items (~1KB/item), the Stripe `payment_intent` response (~3KB), and the enriched Gateway response (~2KB). At 8,000 RPS Black Friday peak, this is ~64 MB/s of JSON logged to CloudWatch. The CloudWatch log ingest cost alone exceeds $0.04/checkout before any compute cost is counted. The per-checkout cost dashboard attributes the cost to "compute" (the log agent CPU) and the real cost driver — log verbosity — is invisible in the container-level cost breakdown. |
| 6 | Identity service auth latency | Latency | Identity Service + Auth0 Management API | p95 < 250ms (token verify at Gateway + profile/consent resolve at Identity Service, cache-hit path) | Synthetic auth-flow probe: `POST /graphql` with a pre-provisioned test-customer JWT, calling `lookupCustomerCart` (which triggers Gateway token validation + Identity Service profile resolution). Probe runs every 60s from 3 regions. Assert: p95 ≤ 250ms over trailing 1,000 probes. Run continuously in staging; gate PR merge if p95 exceeds 250ms in the 1h after deployment | "Customer creates an account on web, can't use it in-store" — if identity resolution is slow, the store associate at the POS waits for the cart to hydrate while the customer stands at the counter. Identity latency is on the critical path of every cart read, not just login | Calling the Auth0 Management API synchronously inside the Cart Service's `getActiveCart` resolver to resolve the customer's loyalty tier for cart pricing. A single cart read triggers: Gateway token validation (Auth0 JWKS, ~20ms), Identity Service profile resolution (local PostgreSQL cache, ~5ms), PLUS an inline Auth0 Management API call for loyalty tier (~100ms p95). The loyalty tier is needed for cart pricing, so the developer added it to the cart resolver rather than reading from the Identity Service's local cache. At 250ms p95 for identity + cart combined, any Auth0 Management API degradation pushes the total past the target. The Gateway and Cart Service latency dashboards both look green (each shows its own segment as healthy); the end-to-end breach is invisible because no single container owns the full 250ms budget. |
| 7 | EU SCA completion & PII boundary | Security & compliance | Checkout Service + Stripe payment gateway + Gateway PII-scrub stage | 100% of EU card payments complete PSD2 SCA; 0 PII fields (PAN, CVV, cardholder name, billing address) outside the PCI trust boundary (Stripe iframe/SDK → Stripe API) | CI control-conformance check: a static grep assertion across all repositories that no API endpoint path, GraphQL field, or database column accepts a raw PAN or CVV. Nightly egress scan: a data-residency check verifies that no log stream, Kafka topic, or database table in Meridian-owned infrastructure contains a string matching a PAN regex. Any match triggers a blocking P1 incident | "PSD2 SCA for EU payments"; "PCI-DSS Level 1" — every EU card payment must route through the SCA challenge flow; no cardholder data may exist outside the PCI-certified boundary (Stripe) | Server-side Stripe integration where the Web App BFF collects the raw card number in a Meridian-controlled `<input>` and `POST`s it to the Apollo Gateway, which forwards it to the Checkout Service, which calls Stripe's `POST /v1/payment_methods` with the raw PAN. The developer argues "we're just proxying — we never store it." But the PAN transits Meridian's TLS termination point (Apollo Gateway) and Meridian's application code (Checkout Service). This expands PCI DSS Level 1 scope to the Gateway, Checkout Service, their deployment pipelines, their log streams, their monitoring agents, and every engineer with production access. The CI grep passes because the PAN field is named `cardNumber` (not on the forbidden list, which only checked for `pan` and `cvv`). The egress scan misses it because the PAN is in transit (TLS-encrypted between containers) and never written to a log file — it's invisible to static analysis, but PCI scope has already expanded. |
| 8 | Redis connection-pool exhaustion under cart-read flood | Reliability | Cart Service → Inventory Read Cache (Redis) | Cart Service Redis client SHALL use connection pooling: maxConnections = 50, maxWaitingClients = 200. When pool exhausted, SHALL degrade to `availability: "unknown"` for remaining items rather than blocking the request. | k6 load test: 5,000 RPS with 5-item carts (25,000 Redis lookups/sec). Assert: zero `Redis connection refused` errors; degraded items marked `availability: "unknown"` with `failureReason: "cache-pool-exhausted"`; p99 cart-read latency remains < 500ms. Run on PR merge to release branch. | "1,400 POS systems" + Black Friday peak — a Redis connection-pool exhaustion during peak turns every product page into a wall of grey "stock unknown" labels. The system stays up (HTTP 200) but is functionally blind. | Configuring the Cart Service's Redis client with `maxConnections = Infinity` or omitting `maxWaitingClients`. Under peak load, the Redis server hits its `maxclients` limit (default 10,000) before the Cart Service's pool limit is reached. The Redis server rejects new connections; the Cart Service retries indefinitely; the Node.js event loop blocks on connection retries; the Gateway times out waiting for Cart Service; the entire cart-read path collapses — all because the pool limit was enforced on the wrong side of the connection (server-side maxclients instead of client-side maxConnections). |
| 9 | SQL injection at loyalty-QR lookup | Security | Identity Service → PostgreSQL (local store) | 100% of database queries in Identity Service SHALL use parameterized queries. QR code format SHALL be validated against regex `^MRD-LOY-[A-Z0-9]{4}-[A-Z0-9]{4}$` before any DB access. PostgreSQL application user SHALL lack DDL privileges (no DROP, ALTER, CREATE, TRUNCATE). | CI static analysis: a grep rule rejects any string concatenation or template literal used to construct a SQL query (`+.*SELECT`, `+.*INSERT`, `+.*DELETE`, `+.*UPDATE`, `${.*}` inside a SQL string). Integration test: scan a QR code containing `MRD-LOY-\'; DROP TABLE--` and assert the Identity Service returns `LOYALTY_QR_UNKNOWN` (format validation rejects it before SQL execution) and the customers table is unchanged. Run on every PR that touches Identity Service. | "strict EU/JP/US regulatory layer" — a SQL injection that drops the customers table is a personal-data breach across all three jurisdictions. GDPR breach notification = 72 hours. APPI breach notification = "without delay." CCPA = 45 days with statutory damages. The QR code is scanned from a piece of paper any customer can print — this is the widest attack surface in the entire platform. | Using an ORM and assuming it prevents injection. ORMs cannot prevent injection when the developer uses raw query methods (`sequelize.query()`, `knex.raw()`, `db.query()`) with string-interpolated input. A developer adds a `resolveLoyaltyQr` function that calls `db.query(\`SELECT * FROM customers WHERE qr_code = '${qrCode}'\`)` — the backtick template literal bypasses the ORM's parameterization. The CI static analysis rule only checks for `+` concatenation, not template literals. |
